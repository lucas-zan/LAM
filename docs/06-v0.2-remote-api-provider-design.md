# LAM v0.2.0 Remote API Provider Design

状态：规划中  
目标版本：v0.2.0  
范围：Codex profile 绑定远程 API Provider；兼容 Responses API 与 `/chat/completions` Provider；保持配置简洁、安全、可解释。

## 1. 背景与约束

LAM v0.1.0 已完成 Codex 多 profile、session、relay、quota 与基础 Provider/Secret 能力。v0.2.0 的核心目标是让用户在添加账号时可以直接选择“远程 API Provider”，并让 Codex 使用该 Provider 工作。

关键约束：

- Codex 当前 provider 配置写在 user-level `$CODEX_HOME/config.toml`，provider 相关配置不应放在 project-local `.codex/config.toml`。
- Codex `model_providers.<id>.wire_api` 当前只支持 `responses`；因此 Codex 侧应始终看到 Responses-compatible endpoint。
- OpenAI Chat Completions 是基于 messages conversation 的接口；如果远程服务只提供 `/chat/completions`，LAM 需要在本机提供 Responses-compatible adapter，再由 adapter 转发到上游 `/chat/completions`。
- Secret 永远不写入 `config.toml`、wrapper、日志、剪贴板或 session 文件；优先 Keychain，其次 env var 引用。

官方依据：

- Codex config reference: `model_provider` 来自 `model_providers`，provider 配置属于 user-level config；`model_providers.<id>.wire_api` 当前支持 `responses`。
- OpenAI Chat Completions overview: `/chat/completions` 从 messages conversation 生成模型响应；官方建议新项目优先考虑 Responses。

## 2. 产品原则

### 配置简洁

用户不需要理解 Codex 的所有 TOML 字段。添加账号时只选择：

- 认证方式：ChatGPT 登录 / Remote API
- Provider 类型：Responses-compatible / Chat Completions-compatible
- Provider：已有 Provider 或新建 Provider
- Model：使用 Provider 默认模型或覆盖
- Secret 存储：Keychain / env var

LAM 负责生成 Codex 可用配置。

### 体验明确

UI 必须明确告诉用户：

- Responses-compatible Provider 可直连 Codex。
- Chat Completions-compatible Provider 需要 LAM 本地 adapter 运行。
- adapter 未运行时，使用该 Provider 的 Codex profile 不应显示为“配置成功”，而应显示“需要启动 adapter”。
- 如果 Provider 不支持工具调用、流式、结构化输出或 reasoning metadata，UI 要展示能力降级。

### 边界清晰

保持三层边界：

- Profile：一个 Codex 工作目录和运行时身份。
- Provider：远程模型服务配置和 secret 引用。
- Adapter：本机协议转换与观测服务，不属于账号数据。

## 3. 目标用户流程

### 添加远程 API 账号

1. 用户点击 Add Account。
2. 选择账号类型：
   - ChatGPT Login
   - Remote API
3. 如果选择 Remote API：
   - 选择已有 Provider，或创建新 Provider。
   - 选择 Provider wire kind：Responses / Chat Completions。
   - 填写 base URL、model、secret 存储方式。
   - LAM 显示将写入哪些 config 字段和是否需要 adapter。
4. 用户确认后：
   - 创建 `$HOME/.codex-<name>`。
   - 写入 `config.toml`。
   - 如果是 Chat Completions Provider，创建 adapter route 并确保 adapter 可启动。
   - 生成 wrapper（如适用）。
5. UI 显示 profile 与 Provider 绑定状态。

### 绑定已有账号到远程 API

1. 用户在 Account detail 或 Provider Center 选择 Attach Provider。
2. LAM 生成 plan：
   - 备份原 `config.toml`。
   - 写入 `model`、`model_provider`、`model_providers.<id>`。
   - 对 Chat Completions Provider 写入本地 adapter base URL，而不是上游 base URL。
3. 执行后，账号卡片显示 Provider、model、adapter 状态。

## 3.1 接力与切换如何兼容 API 账号

LAM 的接力和切换不应绑定在“ChatGPT 登录账号”上，而应绑定在 profile 边界上。只要一个 profile 有可运行的 Codex config，它就可以参与 session 浏览、resume、relay、sync 和切换。

### 基本原则

- Session 仍然属于某个 Codex profile，不属于 Provider。
- Provider 决定该 profile resume 之后用哪个远程模型服务。
- Relay/sync 复制的是 session 上下文，不复制 auth secret。
- API Provider secret 仍由目标 profile 的 Provider 绑定提供。
- 如果 source 和 target 使用不同 Provider，LAM 不阻止接力，但必须显式展示 provider mismatch。

### 切换账号

用户从 ChatGPT 登录 profile 切换到 API profile 时：

```text
source profile: ~/.codex-main       provider=openai/chatgpt
target profile: ~/.codex-api-team   provider=company-proxy
```

LAM 行为：

1. 保留 source session 原始 provider/model 信息。
2. 在 target profile 中执行 resume。
3. Codex 使用 target profile 的 `config.toml` 和 Provider。
4. UI 显示 `original provider -> runtime provider`。
5. 如果模型、工具能力、wire kind 不一致，显示 mismatch / degraded warning。

### 接力 workspace

API profile 可以作为 relay runtime，也可以作为 relay source。

推荐场景：

```text
runtime profile: api-company
source profile: main
relay profile: main-relay-api-company
```

relay profile 的 Provider policy：

- `inherit_runtime`：默认。relay 使用 runtime API Provider。
- `inherit_source`：只在 source 也是 API Provider 且 capability 兼容时允许。
- `explicit`：用户明确选择 Provider。

v0.2.0 默认采用 `inherit_runtime`，因为它最符合“运行身份由目标 profile 决定”的安全边界。

### Chat Completions Provider 的额外要求

如果目标 profile 使用 Chat Completions Provider：

- LAM adapter 必须 running。
- adapter route 必须可解析目标 Provider。
- capability check 必须通过最低要求。
- 不支持的 tool/function/streaming 能力必须在 resume 前展示 degraded warning。

如果 adapter 不可用：

```text
Relay/Resume button: disabled or requires adapter start
Status: Adapter required for this Provider
```

### 当前支持状态

v0.1.0 当前已经具备：

- profile 级 session 浏览。
- profile 级 resume command 生成。
- relay/sync 的 provider mismatch warning 基础。
- Provider CRUD 与 Attach Provider 基础。

v0.1.0 当前不完整：

- Add Account 时直接创建 Remote API profile。
- 官方 `model_providers.<id>` TOML 完整写入。
- Chat Completions adapter。
- API Provider capability check。
- adapter 状态与 Relay/Resume 按钮联动。

因此，当前“已手工 attach 且 Codex config 可用”的 Responses Provider profile 理论上可以参与接力/切换；但产品级体验尚未完成。Chat Completions Provider 必须等 adapter 完成后才能可靠参与接力/切换。

## 4. Provider 模型

v0.2.0 将现有 `ProviderProfile` 扩展为：

```text
id
name
kind: responses | chat_completions
base_url
default_model
env_key
secret_storage: keychain | env | none
health: untested | available | degraded | unavailable

capabilities:
  streaming: supported | unsupported | unknown
  tool_calls: supported | unsupported | unknown
  structured_outputs: supported | unsupported | unknown
  reasoning_metadata: supported | unsupported | unknown

adapter:
  required: bool
  route_id: optional string
  local_base_url: optional string
  upstream_path: /v1/chat/completions | custom
```

### 最小配置与高级配置

Remote API Provider 的最小必填项应保持为三类：

```text
base_url
api_key secret reference
model
```

其中 API key 不作为明文字段保存，而是立即转换成：

```text
secret_storage: keychain | env
env_key: optional string
```

UI 默认只展示最小配置：

- Base URL
- API Key 或 env key
- Model
- API kind：Responses / Chat Completions

高级配置默认折叠：

- Provider ID
- display name
- model list
- default model
- request timeout
- stream timeout
- retry count
- custom headers metadata
- capability overrides

### 多模型策略

Codex 每次运行仍然只能选择一个 active `model`，但一个 Provider 可以保存多个可选模型。

Provider 应支持：

```text
models[]:
  id
  label
  context_window: optional
  supports_tools: unknown/supported/unsupported
  supports_streaming: unknown/supported/unsupported
  supports_reasoning: unknown/supported/unsupported
default_model
```

账号绑定 Provider 时，用户选择该 profile 的 active model：

```text
profile model = provider.default_model or explicit selected model
```

使用体验：

- Add Account 时选择默认模型。
- Account detail 可以切换模型。
- Resume/Relay 前可以临时选择目标模型。
- LAM 写入 `config.toml` 的仍然是一个 active `model`。
- 多模型列表只存在 LAM metadata 中，用于 UI 选择、验证和快速切换。

完全兼容的含义不是把所有模型同时写给 Codex，而是：

- Provider 能保存多个模型。
- Profile 当前运行选择一个模型。
- 切换模型会更新 profile config，并保留备份。
- Relay/Resume 显示 original model -> runtime model。

UI 对用户展示 `kind`，内部仍可保留 `wire_api` 字段用于兼容现有 store，但语义应收敛为：

- `responses`: Codex 直连 Provider。
- `chat_completions`: Codex 连接 LAM adapter；adapter 连接上游 `/chat/completions`。

## 5. Codex config 写入策略

### Responses Provider

LAM 写入：

```toml
model = "<model>"
model_provider = "<provider_id>"

[model_providers.<provider_id>]
name = "<display_name>"
base_url = "<provider_base_url>"
env_key = "<ENV_KEY>"
wire_api = "responses"
```

### Chat Completions Provider

LAM 不直接把上游 `/chat/completions` 配给 Codex。LAM 写入本地 adapter endpoint：

```toml
model = "<model>"
model_provider = "<provider_id>"

[model_providers.<provider_id>]
name = "<display_name>"
base_url = "http://127.0.0.1:<adapter_port>/v1"
env_key = "<LAM_ADAPTER_TOKEN_ENV>"
wire_api = "responses"
```

adapter 再使用 Provider secret 调上游：

```text
POST <upstream_base_url>/chat/completions
Authorization: Bearer <secret from Keychain/env>
```

这样 Codex 始终使用 `responses`，而 LAM 承担协议转换。

## 5.1 API 账号如何启动 Codex

添加 Remote API 账号后，LAM 仍然创建一个普通 Codex profile：

```text
~/.codex-api-company
```

并继续支持 wrapper 命令：

```text
codex-api-company
codex-luna002
codex-c
```

wrapper 的职责：

- 设置 `CODEX_HOME="$HOME/.codex-<name>"`。
- 必要时确保 LAM adapter 已运行，或给出明确错误。
- 执行真实 `codex "$@"`。

Responses Provider 账号：

```text
codex-api-company
```

可以直接启动。

Chat Completions Provider 账号：

```text
codex-api-chat
```

启动前需要满足：

- LAM adapter 正在运行。
- profile config 指向 `http://127.0.0.1:19190/v1`。
- adapter route 能解析该 Provider。

如果 adapter 未运行，wrapper 有两种策略：

1. 最小策略：打印错误并提示打开 LAM。
2. 目标策略：调用 LAM helper 启动 adapter，然后继续执行 Codex。

v0.2.0 建议先实现最小策略，避免 wrapper 在后台静默启动 GUI app。

### 命令兼容

必须保留现有使用方式：

```text
codex-luna002
codex-c
codex-api-company
```

同时 LAM UI 提供 Copy Command：

```bash
CODEX_HOME="$HOME/.codex-api-company" codex
```

对于 Remote API profile，`codex resume <session-id>`、`codex --profile` 等 Codex 原生命令参数应原样透传。

## 6. Adapter 设计

### 职责

本地 adapter 是一个 loopback-only HTTP 服务：

```text
Codex -> POST /v1/responses -> LAM adapter -> POST upstream /chat/completions
```

职责：

- 接收 Responses-style 请求。
- 转换为 Chat Completions messages。
- 转发上游请求。
- 将上游 response/stream 转换为 Responses-style response/stream。
- 屏蔽 secret，不写请求正文到日志。
- 记录低敏指标：provider、model、latency、status、错误类型。

### MVP 转换范围

v0.2.0 只承诺 Codex 可用的最小闭环：

- 文本 input 转 messages。
- system/developer/user/assistant 基本文本角色映射。
- 非流式 response 转换。
- 流式 SSE 基础文本 delta 转换。
- function/tool call 尽量透传到 Chat Completions `tools`，不支持时返回明确 degraded error。

不在 MVP：

- 多模态文件输入完整转换。
- background responses。
- response state 持久化。
- reasoning encrypted context 转换。
- 完整 tool execution 语义模拟。

这些能力必须在 capability check 中体现，不允许静默伪装支持。

## 7. Adapter 生命周期

推荐方案：LAM app 内置 adapter manager。

状态机：

```text
stopped -> starting -> running -> degraded -> stopped
                    -> failed
```

触发：

- App 启动时，如果存在绑定 Chat Completions Provider 的 profile，自动启动 adapter。
- 用户 Attach Provider 后，如 adapter required，立即启动或提示启动。
- 用户手动 Stop Adapter 时，相关 profile 标记为 unavailable。

端口：

- 默认绑定 `127.0.0.1:0` 自动分配端口。
- 实际端口写入 LAM runtime state，不写死到 Provider 永久 metadata。
- profile config 需要稳定 endpoint；因此建议 adapter 使用固定 control port 加 route path，或在启动后重写相关 profile config。

v0.2.0 建议采用稳定端口：

```text
127.0.0.1:19190
```

Provider route：

```text
/v1/responses?provider=<provider_id>
```

如果端口占用，adapter 启动失败并给出明确 UI 错误，不自动随机改 config。

## 8. Secret 策略

Keychain 模式：

- `providers.json` 只保存 `secret_storage = "keychain"`。
- Keychain account：`provider:<provider_id>`。
- adapter 请求上游前读取 secret。

Env 模式：

- `providers.json` 保存 `env_key`。
- adapter 运行时从自身环境读取 env var。
- UI 提供“复制 export 命令”，但不显示 secret。

禁止：

- 明文写入 `config.toml`。
- 明文写入 wrapper。
- 明文传给前端。
- 明文写日志。

## 9. Provider 测试与健康检查

Provider Test 分为两层：

1. Metadata check：
   - base URL 合法。
   - model 非空。
   - secret 引用可解析。
2. Live check：
   - Responses Provider：请求 `/v1/responses` 轻量 smoke。
   - Chat Completions Provider：请求 `/chat/completions` 轻量 smoke，并验证 adapter 转换 smoke。

结果：

```text
available: 可用
degraded: 可用但部分 capability 不支持
unavailable: 不可用
untested: 未测试
```

## 11. 错误与降级

必须结构化错误：

```text
PROVIDER_KIND_UNSUPPORTED
PROVIDER_SECRET_UNAVAILABLE
ADAPTER_NOT_RUNNING
ADAPTER_PORT_IN_USE
ADAPTER_TRANSLATION_UNSUPPORTED
UPSTREAM_CHAT_COMPLETIONS_FAILED
UPSTREAM_RESPONSES_FAILED
```

UI 规则：

- adapter 未运行，不允许显示为 provider available。
- 上游不支持工具调用，标记 degraded。
- 转换失败必须展示“哪类能力不支持”，而不是泛化为网络错误。

## 12. 更友好的 Resume / Relay 执行方式

v0.1.0 的 Resume/Relay 倾向于打开新的 Terminal。v0.2.0 应引入更友好的执行选项，避免用户越用越多窗口。

### 执行模式

提供三种启动模式：

```text
Terminal: 打开系统 Terminal，新建窗口或 tab
Copy command: 只复制命令，用户自己粘贴执行
LAM Runner: 在 LAM 内部启动/管理 Codex 子进程
```

默认建议：

- 普通 Resume：优先 `LAM Runner`。
- 用户需要完整 TTY 体验时：选择 `Terminal`。
- 高风险或调试场景：选择 `Copy command`。

### LAM Runner

LAM Runner 是 v0.2.0 的推荐体验方向：

```text
LAM UI -> Runner service -> codex subprocess -> embedded terminal/log panel
```

能力：

- 一个 profile 一个活动 run。
- 可复用同一个运行面板，不反复开 Terminal。
- 支持 start/stop/restart。
- 显示当前 profile/provider/model/session/cwd。
- 输出只在本机显示，不写入 LAM 日志明文。
- Runner 失败时提供 Copy command fallback。

限制：

- 需要 PTY 支持，否则 Codex TUI 体验可能不完整。
- v0.2.0 可以先实现“managed external process + output panel”，完整交互式 PTY 可进入 v0.2.x。

### Terminal 策略优化

如果仍使用 Terminal：

- 默认打开新 tab，而不是新窗口。
- 对同一个 profile/session 尽量复用已有 Terminal tab 不强求，macOS 自动化能力有限。
- UI 提供设置：`Open in Terminal window/tab`、`Copy command only`、`LAM Runner`。

## 13. 实施顺序

1. Provider model 扩展，不改变现有 attach 行为。
2. Codex config writer 收敛为官方 `model_providers.<id>` TOML。
3. Add Account 支持 Remote API，但只先支持 Responses Provider。
4. Adapter manager 骨架。
5. Chat Completions adapter 非流式文本闭环。
6. Chat Completions adapter 流式文本闭环。
7. Provider capability check 与 UI degraded 状态。
8. Wrapper 命令兼容 API profile。
9. Runner / Terminal 启动体验优化。
10. 手工验证真实 Codex profile 使用 Responses Provider 与 Chat Completions Provider。

## 14. 验收标准

v0.2.0 完成时必须满足：

- 用户可以在 Add Account 中选择 Remote API。
- 用户只填 base URL、API key/env key、model 就能完成最小配置。
- Provider 可保存多个模型，profile 选择一个 active model。
- Responses Provider 可以直连 Codex。
- Chat Completions Provider 可以通过 LAM adapter 给 Codex 使用。
- Remote API profile 仍支持 `codex-<name>` wrapper 命令启动。
- wrapper 参数原样透传给 Codex。
- Codex `config.toml` 不包含 API key 明文。
- Provider secret 不返回前端。
- 绑定 Provider 前有 plan，执行前备份 config。
- adapter 仅监听 loopback。
- adapter 不记录 prompt/body 明文。
- UI 能区分 available/degraded/unavailable。
- API profile 可以作为 relay runtime。
- API profile 可以作为 relay source。
- ChatGPT 登录 profile 与 API profile 之间 resume/sync 时展示 original/current Provider。
- Chat Completions Provider adapter 不可用时，Relay/Resume 明确不可执行或要求先启动 adapter。
- Resume/Relay 至少支持 Copy command fallback，并规划 LAM Runner 或 Terminal tab 体验。
- 测试覆盖 provider model、config writer、secret 边界、adapter 转换、UI 状态。
