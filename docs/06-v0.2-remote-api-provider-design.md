# LAM v0.2.0 Remote API Provider Design

状态：规划中，已按 2026-06-18 调研结果收敛
目标版本：v0.2.0  
范围：Codex profile 绑定远程 API Provider；Responses-compatible Provider 作为稳定能力；Chat Completions Provider 提供 text-only experimental adapter MVP；安全写入 Codex user-level provider config；保持 session/relay 能力按 profile 边界工作。

## 0. 调研结论

v0.2.0 的最佳方案不是一次性完整支持所有 `/chat/completions` Provider，而是 **Responses-first + Chat Completions experimental MVP**：

1. Codex 官方配置已经支持自定义 `model_providers.<id>`，并通过 `model_provider` 选择 Provider。
2. Provider、`openai_base_url`、`model_provider`、`model_providers` 等凭据/Provider 相关配置不能放在项目级 `.codex/config.toml`，必须写入 user-level `$CODEX_HOME/config.toml`。
3. 官方示例中 `wire_api = "responses"` 是自定义 Provider 的可用路径；OpenAI API 文档也建议新 text generation 项目优先使用 Responses API。
4. Codex 官方支持 Provider 的 `env_key` 和 command-backed auth。LAM 应优先使用这些官方机制，而不是把 secret 交给前端、wrapper 或明文 config。
5. Chat Completions -> Responses adapter 的语义风险显著高于直连 Responses Provider：stream event、tool call、reasoning metadata、response state、错误语义都需要转换。因此 v0.2.0 只纳入 text-only、非流式、明确 experimental 的最小闭环；流式、tools、reasoning metadata 进入 v0.2.x。
6. `codex app-server` 是更深的嵌入式集成协议，适合后续 Runner/内嵌 UI；v0.2.0 不应把 LAM Runner 作为 Remote API Provider 的阻塞依赖。

依据：

- Codex Advanced Configuration: <https://developers.openai.com/codex/config-advanced>
  - custom model providers、`openai_base_url`、project config 禁止覆盖 Provider 相关 key、command-backed auth。
- Codex Sample Configuration: <https://developers.openai.com/codex/config-sample>
  - `model_provider`、`model_providers.<id>`、`wire_api = "responses"`、`env_key`、request/stream tuning。
- Codex Environment Variables: <https://developers.openai.com/codex/environment-variables>
  - Provider API key 通过 `env_key` 指定的变量读取。
- Codex App Server: <https://developers.openai.com/codex/app-server>
  - app-server 是 JSON-RPC 深集成协议，非 v0.2.0 Remote API Provider 的必要依赖。
- OpenAI Text Generation docs: <https://developers.openai.com/api/docs/guides/text>
  - 新 text generation app 推荐 Responses API，旧 Chat Completions 不应作为新能力的默认目标。

## 1. v0.2.0 决策

### 1.1 稳定范围

- Provider model 扩展为 Responses-compatible remote API Provider。
- Add Account 支持 `ChatGPT Login` / `Remote API` 分支。
- Remote API profile 写入官方 Codex TOML：
  - `model`
  - `model_provider`
  - `[model_providers.<id>]`
  - `base_url`
  - `env_key` 或 `[model_providers.<id>.auth]`
  - `wire_api = "responses"`
  - 可选 `query_params`、retry/timeout tuning。
- Provider secret 不进入前端状态、`config.toml`、wrapper、日志、session 文件。
- Provider Center 支持 Test/Edit/Attach/Delete，并展示 health、used profiles、model、secret mode。
- Remote API profile 继续参与 session 浏览、resume、sync、relay。
- 不同 Provider/model resume 时展示 original/current mismatch 和 capability warning。
- wrapper 保持 `CODEX_HOME` 隔离并原样透传 `codex "$@"`。
- 对 OpenAI-compatible proxy/router/data residency 场景，优先支持 `openai_base_url` 快捷路径。

### 1.2 实验范围

Chat Completions Provider 在 v0.2.0 必须有可见升级，但只能作为 experimental MVP：

- UI 可创建 `chat_completions` Provider，但必须标记 experimental。
- LAM 内置 loopback-only adapter。
- Codex 侧仍写入 `wire_api = "responses"`，`base_url` 指向本机 adapter。
- adapter 上游调用 `/chat/completions`。
- 只承诺非流式 text-only request/response 闭环。
- 不支持 streaming、tool/function call、structured outputs、reasoning metadata 时必须返回结构化 degraded/unsupported。
- adapter 不记录 prompt/body/secret。
- adapter 未运行时，相关 profile 显示 unavailable，并阻止或提示启动。

### 1.3 范围外

- Chat Completions adapter 的完整 GA 支持。
- Responses stream event 与 Chat Completions SSE 的完整互转。
- tool/function call、reasoning metadata、background responses、response state 的完整模拟。
- LAM Runner / 内嵌 PTY。
- 跨 Agent 通用 Adapter。
- 修改项目级 `.codex/config.toml` 来配置 Provider。

## 2. 产品原则

### Responses-first

LAM v0.2.0 的稳定承诺是 Codex 能直接消费的 Responses-compatible Provider。用户添加 `/chat/completions` Provider 时，UI 应提示：

```text
This provider is Chat Completions-compatible only.
LAM can route basic text requests through an experimental local adapter.
Streaming, tools, structured outputs, and reasoning metadata are not supported in v0.2.0.
```

### 配置简洁

用户只需要理解四件事：

- Account suffix
- Base URL
- Model
- Secret storage: Keychain / env var / auth command

LAM 负责生成 Codex 可用配置。

### 边界清晰

- Profile：一个 Codex 工作目录和运行时身份。
- Provider：远程模型服务配置和 secret 引用。
- Session：属于 profile，不属于 Provider。
- Relay/Sync：复制 session 上下文，不复制 auth secret。

## 3. Provider 模型

v0.2.0 将现有 `ProviderProfile` 扩展为：

```text
id
name
kind: responses | chat_completions
base_url
default_model
models[]
env_key
secret_storage: keychain | env | auth_command | none
auth_command: optional metadata, no secret values
health: untested | available | degraded | unavailable
used_by_profile_ids[]

codex:
  wire_api: responses
  query_params: optional map
  request_max_retries: optional number
  stream_max_retries: optional number
  stream_idle_timeout_ms: optional number

capabilities:
  streaming: supported | unsupported | unknown
  tool_calls: supported | unsupported | unknown
  structured_outputs: supported | unsupported | unknown
  reasoning_metadata: supported | unsupported | unknown

adapter:
  required: bool
  stability: stable | experimental
  local_base_url: optional string
  upstream_path: /v1/chat/completions | custom
```

兼容读取旧 store：

- 旧 `wireApi` 映射为 `kind`。
- `chat_completions` Provider 可以保留在 metadata 中，v0.2.0 UI 标为 experimental。
- 缺少 `models[]` 时，用 `default_model` 补一个模型项。

### 多模型策略

Codex 每次运行只写一个 active `model`。Provider 可以保存多个模型供 UI 选择：

```text
models[]:
  id
  label
  context_window: optional
  supports_tools: unknown/supported/unsupported
  supports_streaming: unknown/supported/unsupported
  supports_reasoning: unknown/supported/unsupported
default_model
```

绑定 profile 时：

```text
profile model = provider.default_model or explicit selected model
```

切换模型必须备份并更新 `$CODEX_HOME/config.toml`。

## 4. Codex Config 写入策略

### 4.1 普通 Responses Provider

```toml
model = "<model>"
model_provider = "<provider_id>"

[model_providers.<provider_id>]
name = "<display_name>"
base_url = "<provider_base_url>"
env_key = "<ENV_KEY>"
wire_api = "responses"
```

约束：

- 不使用保留 Provider ID：`openai`、`ollama`、`lmstudio`、`amazon-bedrock`。
- `env_key` 只写变量名，不写 secret。
- Keychain 模式下，LAM 应生成一个 provider 专属 env var 注入策略，或改用 command-backed auth。

### 4.2 command-backed auth

如果要避免把 API key 放进用户 shell 环境，优先采用 Codex 官方 command-backed auth：

```toml
[model_providers.<provider_id>.auth]
command = "<lam-helper-path>"
args = ["provider-token", "<provider_id>"]
timeout_ms = 5000
refresh_interval_ms = 300000
```

LAM helper 从 Keychain 读取 secret，只向 stdout 输出 bearer token。禁止：

- 输出额外日志到 stdout。
- 接收 stdin。
- 打印 secret 到 stderr 或日志。

### 4.3 OpenAI-compatible proxy 快捷路径

如果用户只想把内置 OpenAI Provider 指到代理、router 或 data residency endpoint，可以使用：

```toml
openai_base_url = "<base_url>"
```

这是一个独立的 Provider mode：

```text
mode: builtin_openai_base_url
```

它不创建 `[model_providers.<id>]`，但仍要遵守 secret 和备份策略。

### 4.4 Azure / query params

Azure 或要求 API version 的服务可写：

```toml
[model_providers.<provider_id>]
name = "Azure"
base_url = "https://YOUR_PROJECT_NAME.openai.azure.com/openai"
env_key = "AZURE_OPENAI_API_KEY"
query_params = { api-version = "2025-04-01-preview" }
wire_api = "responses"
```

### 4.5 Chat Completions Provider experimental config

LAM 不直接把上游 `/chat/completions` 写给 Codex。Codex profile 仍然看到 Responses-compatible endpoint：

```toml
model = "<model>"
model_provider = "<provider_id>"

[model_providers.<provider_id>]
name = "<display_name>"
base_url = "http://127.0.0.1:<adapter_port>/v1"
env_key = "<LAM_ADAPTER_TOKEN_ENV>"
wire_api = "responses"
```

adapter 根据请求路径或 metadata 解析真实 Provider，并使用 Keychain/env/auth command 取上游 secret：

```text
Codex -> POST http://127.0.0.1:<port>/v1/responses
LAM adapter -> POST <upstream_base_url>/chat/completions
```

v0.2.0 不承诺该路径稳定，只承诺 text-only experimental smoke 能跑通。

## 5. 用户流程

### 添加 Remote API Account

1. 用户点击 Add Account。
2. 选择账号类型：
   - ChatGPT Login
   - Remote API
3. Remote API 表单：
   - Account suffix
   - Provider mode：Responses-compatible / OpenAI base URL override
   - Base URL
   - Model
   - Secret storage：Keychain / env var / auth command
   - 可选 query params
4. LAM 展示 plan：
   - create `$HOME/.codex-<name>`
   - write marker
   - write `config.toml`
   - write wrapper
   - store secret reference
   - blocked files: `auth.json`
5. 用户确认后执行。
6. UI 展示 profile、Provider、model、health、session count。

### Attach Provider 到已有 profile

1. 用户在 Account detail 或 Provider Center 选择 Attach Provider。
2. LAM 生成 plan：
   - 备份原 `config.toml`。
   - 写入 `model`、`model_provider`、`model_providers.<id>`。
   - 更新 provider used-by metadata。
3. 执行后刷新账号、Provider、session mismatch 状态。

## 6. Session / Relay 兼容

LAM 的接力和切换绑定在 profile 边界上，不绑定在 ChatGPT 登录账号上。

原则：

- Session 属于 Codex profile。
- Provider 决定 resume 后使用哪个远程模型服务。
- Relay/sync 复制 session，不复制 secret。
- API Provider secret 由目标 profile 的 Provider 绑定提供。
- source 和 target Provider 不一致时不阻止，但必须展示 mismatch。

默认 relay provider policy：

```text
inherit_runtime
```

原因：运行身份和 secret 应由目标 runtime profile 决定。

## 7. Provider Test 与 Health

### Metadata check

- provider ID 合法且不占用保留 ID。
- base URL 合法。
- model 非空。
- secret reference 可解析。
- `wire_api = responses`。

### Config check

- 可生成有效 TOML。
- 写入前能备份。
- 写入后重新解析得到同一 provider/model。

### Live check

Responses Provider：

- 对 `/v1/responses` 发起最小 smoke request。
- 不记录 request body。
- 错误只保留低敏信息：provider、status、错误类型、latency。

Chat Completions Provider experimental：

- 先检查 adapter running。
- adapter 发起最小 `/chat/completions` smoke request。
- adapter 再执行一次 Responses-style text request 到 Chat Completions 的本地转换 smoke。
- 失败时区分 `ADAPTER_NOT_RUNNING`、`ADAPTER_TRANSLATION_UNSUPPORTED`、`UPSTREAM_CHAT_COMPLETIONS_FAILED`。

Health：

```text
available: 可用
degraded: 可用但 capability 未知或部分不支持
unavailable: 不可用
untested: 未测试
experimental: 可用但不属于稳定能力
unsupported: Provider kind 当前不可用
```

## 8. Secret 策略

Keychain 模式：

- `providers.json` 只保存 `secret_storage = "keychain"`。
- Keychain account：`provider:<provider_id>`。
- Codex 通过 LAM auth helper 获取 token，或由 LAM 启动 wrapper 时注入短生命周期 env。

Env 模式：

- `providers.json` 保存 `env_key`。
- Codex 直接读取该变量。
- UI 可以复制 export 命令，但不显示 secret。

禁止：

- 明文写入 `config.toml`。
- 明文写入 wrapper。
- 明文传给前端。
- 明文写日志。
- 明文写入 session/history。

## 10. 错误与降级

结构化错误：

```text
PROVIDER_ID_RESERVED
PROVIDER_KIND_UNSUPPORTED
PROVIDER_SECRET_UNAVAILABLE
PROVIDER_CONFIG_INVALID
PROVIDER_CONFIG_BACKUP_FAILED
UPSTREAM_RESPONSES_FAILED
ADAPTER_NOT_RUNNING
ADAPTER_PORT_IN_USE
ADAPTER_TRANSLATION_UNSUPPORTED
UPSTREAM_CHAT_COMPLETIONS_FAILED
AUTH_HELPER_FAILED
KEYCHAIN_UNAVAILABLE
```

UI 规则：

- Chat Completions Provider 不显示为 stable available。
- Live check 失败不能覆盖 metadata check 的错误细节。
- capability unknown 必须显示为 unknown/degraded，不伪装支持。

## 11. Chat Completions Adapter 后续设计边界

v0.2.0 做 text-only experimental adapter；后续 v0.2.x 才扩展 streaming/tools/reasoning：

```text
Codex -> POST /v1/responses -> LAM adapter -> POST upstream /chat/completions
```

v0.2.0 必须解决：

- Responses request schema 子集定义。
- upstream error 到 Responses error 的规范映射。
- loopback binding、安全 token、日志脱敏。
- adapter route 与 stable profile config 的端口策略。

v0.2.x 再解决：

- Responses streaming event 子集定义。
- tool/function call 支持矩阵。
- reasoning metadata 不可转换时的降级策略。
- 多 choice、复杂 content part、多模态输入。

## 12. 实施顺序

1. Provider schema 迁移：保留旧字段，新增 `kind`、`models[]`、`codex`、`capabilities`。
2. TOML writer/reader：官方 `model_providers.<id>`、`env_key`、auth command、query params、备份恢复。
3. SecretStore：Keychain/env/auth helper 三种策略，测试不泄密。
4. Provider Center v0.2：create/edit/test/attach/delete。
5. Add Account Remote API：默认支持 Responses-compatible；Chat Completions 进入 experimental 分支。
6. Chat Completions adapter feasibility：锁定 text-only schema 子集和错误映射。
7. Adapter manager：loopback-only、start/stop/status、端口冲突错误。
8. Chat Completions 非流式 text-only 转换闭环。
9. Wrapper 兼容：`CODEX_HOME`、参数透传、Copy command fallback。
10. Account/session/relay 状态联动：mismatch、used-by、health、adapter status。
11. 手工验收真实 Responses-compatible Provider。
12. 手工验收真实 Chat Completions Provider experimental adapter。
13. 文档与 roadmap 更新。

## 13. v0.2.0 验收标准

- 用户可以在 Add Account 中选择 Remote API。
- 用户只填 base URL、API key/env key 或 Keychain secret、model 就能完成最小配置。
- Provider 可保存多个模型，profile 选择一个 active model。
- Responses-compatible Provider 可以直连 Codex。
- Chat Completions Provider 可以通过 experimental local adapter 完成一轮普通文本 prompt。
- Codex `config.toml` 使用官方 `model_provider` / `model_providers.<id>` 格式。
- Provider secret 不进入 `config.toml`、wrapper、前端、日志、session/history。
- 绑定 Provider 前有 plan，执行前备份 config。
- 旧 providers.json 可兼容读取或迁移。
- Remote API profile 仍支持 `codex-<name>` wrapper 命令启动。
- wrapper 参数原样透传给 Codex。
- API profile 可以作为 relay runtime。
- API profile 可以作为 relay source。
- ChatGPT 登录 profile 与 API profile 之间 resume/sync 时展示 original/current Provider。
- adapter 仅监听 loopback。
- adapter 不记录 prompt/body/secret。
- adapter 未运行时，Chat Completions profile 明确显示 unavailable，并提示启动 adapter。
- unsupported / experimental Provider kind 有明确 UI 状态。
- 测试覆盖 provider model、config writer、secret 边界、health check、adapter text-only conversion、UI 状态。

## 14. 风险与处理

| 风险 | 处理 |
|------|------|
| 用户填的是 Chat Completions-only Provider | 标记 experimental，只允许 text-only adapter MVP |
| Provider secret 泄露 | Keychain/env/auth helper；日志脱敏；测试覆盖 |
| TOML 写坏导致 Codex 不可用 | 写前备份，写后重新解析，失败回滚或提示恢复 |
| Provider ID 与内置 ID 冲突 | 禁用 `openai`、`ollama`、`lmstudio`、`amazon-bedrock` |
| OpenAI proxy 场景过度建模 | 提供 `openai_base_url` 快捷路径 |
| capability check 不完整 | unknown/degraded，不伪装支持 |
| Chat Completions adapter 拖垮 v0.2 | v0.2.0 只做 text-only 非流式，stream/tools/reasoning 全部移到 v0.2.x |
